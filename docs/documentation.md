# KNPP News Portal (NOT FINAL DOCUMENTATION)

Современный мультиязычный корпоративный портал на **Django + Next.js** с headless-архитектурой.

> Цель проекта — разработать улучшенную версию корпоративного новостного сайта с современным UI/UX, удобной CMS, хорошим SEO, высокой производительностью и возможностью дальнейшего масштабирования.

---

## 1. Обзор проекта

Проект представляет собой корпоративный мультиязычный портал, в котором:

* **Django** отвечает за backend, CMS, admin panel, API и бизнес-логику.
* **Next.js** отвечает за публичный frontend, SEO, производительность и адаптивный интерфейс.
* **PostgreSQL** используется как основная база данных.
* **Redis** используется для кэширования и фоновых задач.
* **S3/MinIO** используется для хранения изображений и документов.
* **Docker** используется для контейнеризации и удобного деплоя.

Архитектура выбрана в формате **headless CMS**, где административная часть и публичный сайт разделены.

---

## 2. Почему выбрана именно такая архитектура

### Основная идея

Система разделяется на две независимые части:

1. **Внутренняя контентная часть** — для редакторов и администраторов.
2. **Публичная часть** — для пользователей, читающих новости и просматривающих разделы сайта.

### Причины выбора

* удобная работа редакторов через Django Admin;
* быстрый и современный frontend на Next.js;
* хорошая индексация страниц поисковыми системами;
* четкое разделение ответственности frontend/backend;
* удобство поддержки и масштабирования;
* возможность в будущем подключить мобильное приложение или другие клиенты к тому же API.

---

## 3. Архитектурный подход

### Headless Architecture

В данной архитектуре:

* **Frontend не хранит контент**, а только запрашивает его через API.
* **Backend не отвечает за внешний рендер интерфейса**, а предоставляет структурированные данные.

### Поток данных

#### Публичный сценарий

`User -> Next.js -> Django API -> PostgreSQL / Media Storage -> Next.js -> User`

#### Редакционный сценарий

`Editor -> Django Admin -> PostgreSQL / Media Storage -> Django API -> Next.js -> Public Website`

## 5. Ключевые преимущества стека

### Почему Django

* встроенная и мощная admin panel;
* быстрое создание CMS-функционала;
* удобная работа с ролями, контентом, документами и CRUD-логикой;
* зрелый и надежный framework для корпоративных решений.

### Почему Next.js

* поддержка **SSR, SSG, ISR**;
* сильные возможности для SEO;
* удобная работа с динамическими маршрутами;
* отличный пользовательский опыт;
* хорошая производительность и современная экосистема.

### Почему PostgreSQL

* подходит для структурированных данных;
* хорошо работает со связями между сущностями;
* надежен и масштабируем;
* отлично интегрируется с Django.

### Почему Redis

* ускоряет часто используемые запросы;
* разгружает backend;
* подходит для фоновых задач и временных данных.

### Почему Object Storage

* удобно хранить изображения, PDF и документы;
* проще масштабировать, чем локальное файловое хранилище;
* удобно использовать вместе с CDN.

## 9. Workflow публикации

Для контроля качества контента используется публикационный процесс.

### Статусы материалов

* `draft` — черновик
* `review` — на проверке
* `published` — опубликовано
* `archived` — архивировано

### Сценарий публикации

1. Редактор создает материал.
2. Материал сохраняется как `draft`.
3. После завершения редактирования он переводится в `review`.
4. Проверяющий или администратор проверяет материал.
5. После одобрения материал переводится в `published`.
6. Материал становится доступен в публичном API.
7. Frontend отображает материал на сайте.

---

## 10. Сценарии использования

### 10.1 Просмотр главной страницы

1. Пользователь открывает сайт.
2. Запрос поступает на Next.js frontend.
3. Frontend запрашивает данные у Django API.
4. Backend возвращает:

   * featured news
   * latest news
   * events
   * documents
   * vacancies
5. Next.js рендерит страницу и показывает ее пользователю.

### 10.2 Просмотр отдельной новости

1. Пользователь открывает карточку новости.
2. Next.js открывает маршрут `/news/[slug]`.
3. Frontend запрашивает новость по slug через API.
4. Django получает данные из PostgreSQL.
5. Backend возвращает контент, SEO-данные и связанные материалы.
6. Frontend отображает страницу новости.

### 10.3 Создание новости редактором

1. Редактор входит в Django Admin.
2. Создает новость.
3. Заполняет заголовок, описание, текст, категорию, теги и SEO-поля.
4. Загружает изображение.
5. Сохраняет материал как `draft`.

### 10.4 Публикация новости

1. Проверяющий открывает материал в статусе `review`.
2. Проверяет корректность данных.
3. Переводит статус в `published`.
4. Материал становится доступным через API.
5. Frontend показывает его на сайте.

### 10.5 Поиск

1. Пользователь вводит поисковый запрос.
2. Frontend отправляет запрос на `/api/v1/search`.
3. Backend ищет совпадения по новостям, документам, страницам и другим сущностям.
4. Результаты возвращаются на frontend.
5. Пользователь видит отсортированный список материалов.

### 10.6 Мультиязычность

1. Пользователь выбирает язык: `ru`, `kk`, `en`.
2. Frontend меняет маршрут.
3. Frontend передает язык в API.
4. Backend отдает материалы нужной языковой версии.
5. Пользователь видит контент на выбранном языке.

## 12. SEO Strategy

Так как сайт является новостным и публичным, SEO — один из ключевых факторов.

### SEO-функции

* server-side rendering для важных страниц;
* ISR/SSG для контентных страниц;
* `sitemap.xml`;
* `robots.txt`;
* canonical URLs;
* Open Graph и Twitter Cards;
* schema.org (`Organization`, `NewsArticle`, `BreadcrumbList`);
* `hreflang` для разных языков;
* генерация метаданных на уровне страницы.

### Почему это важно

* страницы лучше индексируются;
* новости появляются в поиске быстрее;
* улучшается вид страниц при публикации ссылок в мессенджерах и соцсетях.

## 13. Мультиязычность

Сайт поддерживает три языка:

* **Русский** (`ru`)
* **Казахский** (`kk`)
* **Английский** (`en`)

### Подход к хранению переводов

Для каждого материала создаются отдельные записи по языкам, связанные через `translation_group_id`.

### Преимущества такого подхода

* удобство для редакторов;
* лучшее SEO;
* независимая публикация языковых версий;
* простота API и маршрутизации.

### Пример маршрутов

* `/ru/news/...`
* `/kk/news/...`
* `/en/news/...`

* изображения для новостей;

* вложения закупок;
* галереи;

В объектном хранилище: **S3-compatible storage** или **MinIO**.

* проще масштабировать;
* безопаснее;

* удобно подключать CDN;
* нет привязки к одному серверу.

## 18. Режимы рендера Next.js

Для разных страниц используются разные режимы:

### SSG

Подходит для редко меняющихся страниц:

* about
* contacts
* static pages

### ISR

Подходит для:

* homepage
* news list
* article pages
* events

### SSR

Подходит для:

* search results
* сложная фильтрация
* персонализированные или очень динамичные данные

### Почему это важно

Позволяет оптимально балансировать между:

* скоростью;
* SEO;
* свежестью данных;
* нагрузкой на сервер.

## 20. Запуск проекта локально

### Требования

* Docker
* Docker Compose
* Node.js 20+
* Python 3.12+
* PostgreSQL 16+
* Redis 7+

### Вариант через Docker Compose

```bash
git clone <repository-url>
cd knpp-news-portal
cp .env.example .env
docker compose up --build
```

### После запуска

* Frontend: `http://localhost:3000`
* Backend API: `http://localhost:8000/api/v1/`
* Django Admin: `http://localhost:8000/admin/`

### Создание суперпользователя

```bash
docker compose exec api python manage.py createsuperuser
```

### Применение миграций

```bash
docker compose exec api python manage.py migrate
```

### Сбор статики

```bash
docker compose exec api python manage.py collectstatic --noinput
```

## 24. Итог

Данный проект реализует современный подход к разработке корпоративного мультиязычного портала компании.

Выбранная архитектура позволяет:

* создать удобную CMS для редакторов;
* обеспечить быстрый и красивый пользовательский интерфейс;
* поддерживать как корпоративные страницы, так и новостные и сервисные разделы;
* добиться хорошего SEO;
* упростить поддержку и развитие проекта;
* подготовить систему к дальнейшему росту.

**Итоговое архитектурное решение:**

* **Django** — backend, CMS, admin, API, business logic
* **Next.js** — public frontend, SEO, UX, rendering
* **PostgreSQL** — main database
* **Redis** — caching and background jobs
* **S3/MinIO** — media and file storage
* **Docker + Nginx** — deployment infrastructure

На первом этапе ядро backend строится вокруг модулей **pages**, **news** и **vacancies**, потому что они лучше всего соответствуют структуре корпоративного сайта с разделами компании, новостями и карьерным блоком.
